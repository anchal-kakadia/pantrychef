# PantryChef — AI assistant guide

PantryChef is a mobile-first pantry management and recipe recommendation
backend built in Python. Users track pantry items, get LLM-inferred expiry
dates, receive daily expiry email alerts, and request streaming AI recipe
suggestions personalised to their dietary preferences.

The mobile frontend (React Native via Expo) is planned for Week 4 of the
build plan. Right now the focus is the backend, AI layer, evals, and CI.

## Stack

- Python 3.12, FastAPI, async SQLAlchemy 2.0 with asyncpg driver
- PostgreSQL 16 (Docker locally, host port 5433 to avoid conflict with
  another local Postgres on 5432)
- Redis 7 (Docker locally, default port 6379)
- Alembic for migrations
- pytest + pytest-asyncio (mode=auto) for tests
- OpenAI API — gpt-3.5-turbo for shelf-life inference, gpt-4o for recipes
- Celery + Redis for background jobs (Week 2 of plan, not yet implemented)
- AWS SES for transactional email (Week 4 of plan)

## Project structure

- `app/` — all application code; always import as `from app.x import y`
- `app/models/models.py` — SQLAlchemy ORM models (User, PantryItem,
  ShelfLifeCache, NotificationLog)
- `app/schemas/` — Pydantic request/response schemas, one module per domain
- `app/routers/` — FastAPI route handlers, kept thin (delegate to services)
- `app/services/` — business logic, where real work happens
- `app/config.py` — Pydantic Settings reading from .env
- `app/database.py` — async engine, session factory, `get_db` dependency
- `tests/` — pytest, one test file per feature area
- `tests/conftest.py` — shared fixtures including the test `client` fixture
- `alembic/` — migrations, autogenerated then human-reviewed before commit

## Conventions that matter

- SQLAlchemy 2.0 `Mapped[]` syntax everywhere; never legacy `Column()`
- All DB access is async: `AsyncSession`, `await db.execute(select(...))`
- Pydantic schemas reject unknown fields via `model_config = {"extra": "forbid"}`
- PATCH endpoints use `model_dump(exclude_unset=True)` so unsent fields are
  preserved, not nulled
- Ownership checks live in service queries (`WHERE user_id = ?`), not in
  Python code after fetch — never trust an item ID alone
- Soft deletes via `deleted_at` timestamp, never hard delete pantry items
- Use bcrypt directly for password hashing; passlib has version conflicts
  with current bcrypt and was removed deliberately
- Tests mock OpenAI calls via an autouse fixture; never hit real API in CI
- Async DB tests use a fresh engine per test (see tests/conftest.py)
  because asyncpg connections are bound to the event loop they were
  created in — session-scoped engines cause "another operation in progress"
  errors

## What you can do

- Read any file in the repo, suggest changes, run tests with `make test`
- Create migrations with `alembic revision --autogenerate -m "msg"`
- Add new endpoints, services, schemas, and tests
- Refactor code as long as all tests still pass
- Update CLAUDE.md itself if a convention is missing or has changed

## What you should not do

- Push directly to `main` — always work on a feature branch
- Hard delete database rows (use the soft delete pattern)
- Hardcode secrets or commit `.env`
- Add new top-level dependencies without justifying in the PR description
- Modify a migration that has already been applied (write a new one instead)
- Skip running `make test` after changes — broken tests reaching `main` is
  the failure mode this whole setup is designed to prevent

## Verification loop

After making changes, always run in this order. The order matters because
the cheapest check should fail first:

1. `make lint` — ruff catches syntax and import issues in seconds
2. `make test` — full pytest suite (currently 24 tests, ~10 seconds)
3. If touching DB schema, verify `alembic upgrade head` runs clean on a
   fresh DB

If any step fails, fix it before considering the task done. Do not ask me
to merge code with failing tests.

## Things I am intentionally NOT doing

These are decisions, not oversights:

- Not using passlib — incompatible with latest bcrypt, removed deliberately
- Not using requirements.txt — pyproject.toml is the modern standard
- Not deploying to production yet — that's Week 4 of the plan
- Not adding a web frontend — mobile-only via React Native Expo, because
  the camera scan feature is the killer use case and nobody scans
  groceries on a laptop
- Not caching recipe results — pantry state changes constantly, cache
  would be wrong more often than right
